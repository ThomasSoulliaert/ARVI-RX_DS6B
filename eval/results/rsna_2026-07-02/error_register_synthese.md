# Synthèse du registre d'erreurs (§13.5) — run `rsna_2026-07-02`

Registre : `error_register.csv` (28 cas, générés par `eval/build_error_register.py` depuis les CSV de prédictions des **trois** prompts, sélection déterministe). Composition : **baseline** — 1 FN, 3 FP, 6 UA (3 opacités, 3 normales), 4 OK de référence ; **improved** — 4 CM (effondrement par ancrage few-shot) ; **improved_v2** — 5 FP (coût de spécificité), 3 FN à haute confiance, 2 OK récupérés. Le code CM est une extension documentée de la taxonomie FN/FP/UA/JF/HT : il désigne une erreur imputable au comportement induit par le prompt (et non à la perception de l'image). Aucun cas JF (taux de JSON valide = 100 % sur les trois prompts). La revue HT (hallucinations textuelles) est menée à part par `eval/build_ht_review.py` sur les justifications complètes (`justifications_review.csv`).

## Top 5 des causes d'erreur

1. **Ancrage few-shot du prompt improved — puis correction par v2 (finding central).** Le prompt improved ne produit **aucune** prédiction `suspected_opacity` sur 170 cas : sensibilité opacités 0,85 → 0,00. Les 72 vrais positifs baseline régressent en `uncertain` avec une confiance quasi constante de 0,40 (copie du 0,45 de l'exemple few-shot C), sous le seuil de garde-fou 0,60 ; 3 faux négatifs apparaissent à confiance 0,75 (copie du 0,72 de l'exemple A). Le prompt **improved_v2** retire les confiances imitables des exemples : il récupère **81 opacités** que improved envoyait en `uncertain` (sensibilité 0,00 → **0,95**). C'est la preuve que la régression venait bien de la formulation du prompt, pas de la perception.

2. **improved_v2 supprime totalement `uncertain` → le garde-fou est court-circuité (nouveau point de vigilance).** Le prompt v2 ne renvoie **jamais** `uncertain` (taux 0 %) et sort ses confiances de façon quasi-discrète (0,70 / 0,80 / 0,95). Conséquence : ses 4 faux négatifs sortent tous à **0,95**, très au-dessus du seuil `conf < 0,60 → uncertain` — le filet de sécurité ne se déclenche jamais. Gagner en accuracy en supprimant la classe de sécurité **retire le garde-fou** : c'est un compromis à documenter explicitement (les consignes rappellent qu'`uncertain` est un garde-fou, pas un échec). Action : réintroduire une bande d'incertitude ou une vraie calibration de confiance.

3. **Chute de spécificité de improved_v2 (le coût de la correction).** En mettant fin à la sur-prudence, v2 sur-appelle l'opacité sur les normales : **19 FP sur 85 normales**, spécificité 0,96 → 0,78, à confiance élevée (0,70–0,85). Le gain de sensibilité (+0,10 vs baseline) se paie d'une perte de spécificité (−0,18). Action : relire ces FP (revue HT pour détecter une observation inventée) et/ou ajuster le seuil de décision.

4. **Sur-prudence baseline sur les images normales (coût en couverture).** 24 normales sur 85 (28 %) sont renvoyées `uncertain` par la baseline à confiance 0,30–0,50 : premier poste du taux d'incertitude global (21 %) et raison pour laquelle la couverture tombe à 134/170. improved_v2 élimine ce coût (couverture 170/170) mais bascule dans l'excès inverse (FP). Le bon réglage est entre les deux.

5. **Faux négatif persistant sur les trois prompts (cas d'école).** `RSNA_bfb32559` (opacité → `normal`) est raté par baseline (conf 0,80), improved (`normal`) **et** improved_v2 (`normal`, conf **0,95**) : non seulement aucun prompt ne le rattrape, mais la confiance de l'erreur **augmente**. C'est la limite structurelle d'un filet de sécurité fondé sur un seuil de confiance — une erreur affirmée avec assurance passe toujours au travers. À relire visuellement en priorité et à montrer en soutenance.
