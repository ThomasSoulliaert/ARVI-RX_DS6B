  1. C'est quoi un VLM (Vision-Language Model) ?

  Un VLM, c'est un modèle qui sait lire deux choses en même temps : une image et du texte, et qui produit du 
  texte en réponse.

     IMAGE (ta radio)  ┐
                       ├──►  [ VLM : MedGemma ]  ──►  TEXTE (la réponse)
     TEXTE (ton prompt)┘

  C'est l'évolution d'un LLM classique (type ChatGPT, qui ne lit que du texte) : on lui a ajouté des « yeux ».
  MedGemma, c'est exactement ça, mais entraîné sur des données médicales.

  2. Comment l'image entre dans le modèle (le point clé)

  Un modèle ne comprend pas des pixels directement. Il y a une traduction en 3 temps :

  1. L'encodeur d'image (vision encoder) découpe ta radio en petits carrés (patches), et transforme chaque carré
  en une liste de nombres (un vecteur). Ton image devient une séquence de « jetons visuels » (visual tokens).
  2. Le texte de ton prompt est lui aussi découpé en tokens (morceaux de mots) et transformé en nombres.
  3. Les deux séries de nombres sont collées bout à bout et envoyées dans le même cerveau (le transformer). Pour
  le modèle, ton image et ton texte forment une seule longue phrase mélangée.

  ▎ 💡 L'idée à retenir : image et texte finissent dans le même format (des nombres), donc le modèle peut «
  ▎ raisonner » sur les deux ensemble. Quand tu écris dans le prompt « décris l'opacité dans le poumon », il
  ▎ peut relier ces mots aux zones de l'image.

  3. Comment il produit la réponse

  Le modèle génère le texte un token à la fois, en prédisant à chaque fois « quel est le mot le plus probable
  ensuite ? », en boucle, jusqu'à la fin.

  C'est pour ça que deux points sont délicats dans ton projet :
  - Il ne « décide » pas une classe d'un coup : il écrit du texte. Si tu lui demandes du JSON, il va essayer
  d'écrire du JSON, mais il peut se tromper (virgule en trop, blabla avant le {…). → C'est exactement pourquoi
  tes garde-fous existent.
  - Le « score de confiance » qu'il met dans le JSON, c'est un nombre qu'il écrit lui-même, pas une vraie
  probabilité mathématique fiable. À traiter avec prudence (un point à dire à la soutenance).

  4. Le « processor » et le « chat template »

  Dans le code (Hugging Face transformers), tu ne manipules pas les pixels à la main. Un objet processor fait
  tout le sale boulot :

  processor(image + prompt formaté)  →  nombres prêts pour le modèle
  model.generate(...)                →  nombres de réponse
  processor.decode(...)              →  texte lisible

  Le chat template, c'est le format de conversation que MedGemma attend. On ne lui envoie pas du texte brut,
  mais une structure « rôle/contenu » :

  messages = [
    {"role": "user", "content": [
        {"type": "image", "image": ma_radio},
        {"type": "text",  "text": mon_prompt}   # ← ton baseline_prompt.txt
    ]}
  ]

  C'est là que ton prompt (le fichier prompts/baseline_prompt.txt) entre en jeu : c'est le text que tu colles à
  côté de l'image.

  6. Pourquoi « baseline vs improved » a du sens maintenant

  Même image, même modèle — mais tu changes uniquement le prompt :
  - baseline = consigne simple → le modèle est parfois trop sûr de lui, invente des trucs (hallucination),
  oublie le JSON.
  - improved = consigne renforcée (« si doute → uncertain », « ne mentionne rien qui n'est pas visible », « JSON
  strict ») → moins d'hallucinations, plus de uncertain justifiés.

  Tu mesures la différence avec les métriques de la couche 3. C'est ça, « l'amélioration mesurée » qui vaut 20%
  : prouver chiffres à l'appui que le prompt amélioré réduit les erreurs.
